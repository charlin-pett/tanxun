## 记忆条目

### 2026-07-07 — Next.js 16 兼容性问题

**问题：** create-next-app 默认创建的是 Next.js 16，与网上常见的 Next.js 14/15 教程 API 不同。
**具体差异：**
1. `middleware.ts` → 改为 `proxy.ts`（Next.js 16 新规范，不兼容旧版本）
2. next-intl v4 的 `createSharedPathnamesNavigation` → 改为 `createNavigation`
3. `getRequestConfig` 必须包含 `locale` 字段

**解决方案：**
- 构建时看编译报错，通过查看 `node_modules` 中对应包的源码确认正确的 API
- 不要直接套用网上教程，以项目实际安装的版本为准

### 2026-07-07 — 项目命名规范

**问题：** `create-next-app` 不能使用中文项目名（"探寻"）
**原因：** npm 命名规则限制，只能包含 URL 友好字符
**解决方案：** 代码项目名用 `tanxun` 初始化，放在 `D:\探寻` 父目录下即可

---

### 2026-07-07 — 文件读写安全

**问题：** 对已存在的文件直接 Write 会报错
**原因：** 权限模型要求先 Read 已存在的文件才能 Write
**解决方案：** 新文件可直接 Write；已存在的文件先 Read 再 Write 覆盖

---

### 2026-07-07 — 所有源码必须放在 src/ 目录下

**问题：** 在项目根目录下创建 `lib/`、`data/`、`types/`、`engine/`、`prompts/` 等目录，构建时报 `Module not found` 错误。
**原因：** Next.js 默认的 `@/` 路径映射指向 `./src/` 目录，不在 src/ 下的文件无法用 `@/` 导入。
**解决方案：** 所有项目相关源码统一放在 `src/` 目录下，根目录只保留配置文件（next.config.ts, package.json, tsconfig.json 等）和公共目录（messages/, public/）。

### 2026-07-07 — 八字排盘引擎架构

**架构设计思路：**
- `engine/types.ts` — 纯数据结构定义，语言无关
- `engine/ganzhi.ts` — 底层计算（无业务逻辑），语言无关
- `engine/bazi.ts` — 排盘编排（组合底层函数），语言无关
- `engine/labels.ts` — 文字转换（唯一依赖中文的文件，后续加其他语言时替换此文件即可）

**日干支计算方法：** 以1900年1月1日（甲戌日，六十甲子索引10）为基准，计算目标日期到基准日的天数差，mod 60得到六十甲子索引。

**注意：** 月起算和年起算未精确按节气调整（使用近似值），后续可根据需要精确化。
**起运年龄：** 当前固定为3岁（精确计算需用出生日到节气日天数÷3）。

### 2026-07-07 — Next.js 16 兼容性问题

**问题：** create-next-app 默认创建的是 Next.js 16，与网上常见的 Next.js 14/15 教程 API 不同。
**具体差异：**
1. `middleware.ts` → 改为 `proxy.ts`（Next.js 16 新规范，不兼容旧版本）
2. next-intl v4 的 `createSharedPathnamesNavigation` → 改为 `createNavigation`
3. `getRequestConfig` 必须包含 `locale` 字段

**解决方案：**
- 构建时看编译报错，通过查看 `node_modules` 中对应包的源码确认正确的 API
- 不要直接套用网上教程，以项目实际安装的版本为准

### 2026-07-07 — 项目命名规范

**问题：** `create-next-app` 不能使用中文项目名（"探寻"）
**原因：** npm 命名规则限制，只能包含 URL 友好字符
**解决方案：** 代码项目名用 `tanxun` 初始化，放在 `D:\探寻` 父目录下即可

### 2026-07-07 — 文件读写安全

**问题：** 对已存在的文件直接 Write 会报错
**原因：** 权限模型要求先 Read 已存在的文件才能 Write
**解决方案：** 新文件可直接 Write；已存在的文件先 Read 再 Write 覆盖

### 2026-07-07 — 所有源码必须放在 src/ 目录下

**问题：** 在项目根目录下创建 `lib/`、`data/`、`types/`、`engine/`、`prompts/` 等目录，构建时报 `Module not found` 错误。
**原因：** Next.js 默认的 `@/` 路径映射指向 `./src/` 目录，不在 src/ 下的文件无法用 `@/` 导入。
**解决方案：** 所有项目相关源码统一放在 `src/` 目录下，根目录只保留配置文件（next.config.ts, package.json, tsconfig.json 等）和公共目录（messages/, public/）。

### 2026-07-07 — 八字排盘引擎架构

**架构设计思路：**
- `engine/types.ts` — 纯数据结构定义，语言无关
- `engine/ganzhi.ts` — 底层计算（无业务逻辑），语言无关
- `engine/bazi.ts` — 排盘编排（组合底层函数），语言无关
- `engine/labels.ts` — 文字转换（唯一依赖中文的文件，后续加其他语言时替换此文件即可）

**日干支计算方法：** 以1900年1月1日（甲戌日，六十甲子索引10）为基准，计算目标日期到基准日的天数差，mod 60得到六十甲子索引。

**注意：** 月起算和年起算未精确按节气调整（使用近似值），后续可根据需要精确化。
**起运年龄：** 当前固定为3岁（精确计算需用出生日到节气日天数÷3）。

### 2026-07-07 — Turbopack 动态 import JSON 问题

**问题：** 使用 `await import(\`@/data/knowledge/${locale}/${file}.json\`)` 动态导入 JSON 文件时，只有 `yijing.json` 能加载成功，其他 JSON 文件全部 404。
**原因：** Next.js 16 使用 Turbopack 进行开发编译，动态路径的 `import()` 只能识别到实际被引用的文件。`yijing.json` 因为在同一目录下最先被遍历，所以被包含进打包结果，其他文件未被打包。
**解决方案：** 改为**静态导入**——在顶层用 `import data from './file.json'` 一次性导入所有 JSON 文件，然后通过对象映射访问。这样可以确保所有文件都被打包。

### 2026-07-07 — JSON 中文引号转义问题

**问题：** 知识库 JSON 文件中大量使用了中文双引号（如"群经之首"），这些引号与 JSON 的结构化双引号冲突，导致 JSON 解析失败。
**解决方案：** 
- 方案一：使用「」替代中文双引号（推荐，最简洁）
- 方案二：用 `\"` 转义内容中的双引号
- 方案三：通过 `node -e "JSON.stringify()"` 等程序化方式生成 JSON，避免手写

### 2026-07-08 — 解梦功能架构

**架构设计：**
- `engine/dream/categories.ts` — 纯前端分类引擎，支持多语言关键词匹配。7大分类可覆盖多数常见梦境。
- 分类只做关键词匹配，不做 NLP，确保快速且无需 AI。
- 解梦 API 沿用与命理报告相同的双模型策略，四种语言使用独立 prompt 模板。
- Mock 模式根据梦境关键词（水/火/飞/坠落）生成差异化的解梦内容，避免太笼统。

**语言支持：**
- 解梦 API 的四种语言的 Mock 内容各自独立编写，有各自的风格和结构。
- 四种语言的结构板块不同：中文用【古籍解读】、英文用【Ancient Wisdom】、俄文用【Древняя мудрость】、西文用【Sabiduría Antigua】。

### 2026-07-08 — AdSense 验证需要原生 script 标签

**问题：** Next.js 的 `<Script>` 组件渲染后 AdSense 验证爬虫检测不到。
**解决方案：** 改用原生 `<script async>` 标签写在 `<head>` 里。
**经验：** Google 验证爬虫只看原始 HTML，不执行 JS 框架渲染。

### 2026-07-08 — 组件数据也要按语言切换

**问题：** DailyFortune 组件标题翻译了，但卦名加载的中文数据。
**解决方案：** 按 `useLocale()` 加载对应语言的数据文件。
**原则：** 组件显示的数据必须跟随语言切换，不能硬编码。
